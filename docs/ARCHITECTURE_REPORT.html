📊 Executive Summary

This document provides a comprehensive technical analysis of the Debt Collection Voice Agent system. The application enables real-time voice conversations between a debt collection AI agent and a user playing the role of a debtor. The system uses LiveKit Cloud for real-time audio, Fish Audio for text-to-speech, and GPT-5.2 via LiveKit Inference for natural language understanding.

Key Achievement: Full-stack voice AI application deployed to production in 6 phases, from static landing page to business logic with commitment detection.

🧩 Core Component Analysis

1. Token API (/api/token/route.ts)

Purpose

Generates short-lived JWT tokens for browser authentication with LiveKit Cloud.

Key Responsibilities

  • Creates unique room names (demo-{uuid})
  • Assigns participant identities (user-{uuid})
  • Grants permissions: roomJoin, canPublish, canSubscribe, canPublishData
  • Configures automatic agent dispatch via RoomAgentDispatch

Security

// Token expires in 10 minutes (demo session is short) const at = new AccessToken(API_KEY, API_SECRET, { identity, ttl: "10m" }); // Auto-dispatch agent when room is created at.roomConfig = new RoomConfiguration({ agents: [new RoomAgentDispatch({ agentName: "agent" })] });
Why This Matters: The RoomAgentDispatch ensures the agent is automatically dispatched when the room is created. Without this, the user would need to manually trigger agent dispatch.

💼 Business Logic: Debt Collection Flow

Agent Persona

The agent is configured with a specific persona and behavioral constraints:

You are Alex, a debt collection specialist at Acme Finance. Your goal: Get the debtor to commit to a specific payment amount and date. Rules: - Keep responses short (1-2 sentences max) - Ask one question at a time - Be professional but firm - When the debtor agrees to pay a specific amount on a specific date, call the record_repayment_commitment tool with that amount and date - After recording, confirm the commitment and end the call politely Always reply in English.

Function Tool: Commitment Detection

The record_repayment_commitment function tool enables the LLM to:

  1. Parse commitment details: Extract amount and date from conversation
  2. Send structured event: Publish JSON to data channel
  3. Provide feedback: Return confirmation message to LLM

Why Function Tools?

Function Tools vs. Regex Parsing:
✅ Reliable: LLM outputs structured data via function call API
✅ Context-aware: LLM understands conversation context
✅ Extensible: Easy to add more tools (e.g., payment processing)
❌ Regex parsing is fragile and doesn't scale

🎯 Key Technical Decisions

1. Why LiveKit Inference for STT/LLM?

Decision: Use LiveKit Inference instead of direct API keys for Deepgram and OpenAI.

Rationale:
• ✅ No API keys to manage for STT/LLM
• ✅ Integrated with LiveKit Agents framework
• ✅ Free tier available for demos
• ✅ Lower latency (co-located with SFU)

Trade-off: Less control over model selection and pricing at scale.

2. Why Fish Audio for TTS?

Decision: Use Fish Audio instead of LiveKit Inference (Cartesia) or ElevenLabs.

Rationale:
• ✅ Hard constraint from project requirements
• ✅ Custom voice cloning capabilities
• ✅ Streaming support for low latency
• ✅ Competitive pricing

Trade-off: Requires separate API key and network access (blocked in some corporate networks).

3. Why Function Tools Instead of Regex?

Decision: Use @function_tool decorator for commitment detection.

Rationale:
• ✅ LLM natively understands structured outputs
• ✅ Type-safe parameter extraction
• ✅ Built-in validation by LiveKit Agents
• ✅ Easy to extend (add more tools)

Alternative Considered: Regex parsing of LLM text output.
Rejected Because: Fragile, hard to maintain, no semantic understanding.

4. Why Data Channel Instead of Transcripts?

Decision: Use LiveKit data channel for commitment events, not transcript parsing.

Rationale:
• ✅ Structured JSON data (no parsing needed)
• ✅ Real-time delivery (sub-100ms)
• ✅ Independent of transcript accuracy
• ✅ Can send any event type

Trade-off: Requires agent-side implementation (can't use off-the-shelf agent).

🔒 Security Considerations

Secrets Management

  • ✅ No hardcoded secrets: All API keys in environment variables
  • ✅ .env files ignored: .env.local in .gitignore
  • ✅ Short-lived tokens: 10-minute TTL for LiveKit tokens
  • ✅ Secret injection: LiveKit Cloud manages secrets securely

Access Control

  • ✅ Room isolation: Each session gets unique room
  • ✅ Token scopes: Limited permissions (can't delete rooms, etc.)
  • ✅ No cross-user access: Random UUIDs prevent URL guessing

Data Privacy

  • ✅ No persistence: No database, no recordings stored
  • ✅ Ephemeral rooms: Rooms deleted after session ends
  • ✅ Fictional data: All debtor information is fake (demo purposes)

⚡ Performance Characteristics

Latency Breakdown

TTS → User speaker 100-200ms WebRTC propagation
Total (end-to-end) 1-2.5 seconds First audio byte to user

Optimization Techniques

  • Preemptive Generation: LLM starts generating before turn ends
  • Streaming TTS: Audio sent as soon as first chunk ready
  • VAD Caching: Silero model loaded once at startup (prewarm)
  • Adaptive Stream: LiveKit adjusts quality based on network
